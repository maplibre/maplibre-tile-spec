# mlt-wasm

WebAssembly bindings for the [MapLibre Tile (MLT)](https://github.com/maplibre/maplibre-tile-spec) decoder.

This crate compiles `mlt-core` to WebAssembly via [wasm-bindgen](https://github.com/rustwasm/wasm-bindgen) and ships a small TypeScript wrapper that exposes a clean, lazy `VectorTileLike` API.

## Structure

```
mlt-wasm/
├── src/
│   └── lib.rs       # Rust — wasm-bindgen bindings, compiled by wasm-pack
├── js/
│   ├── index.ts     # Package entry point
│   └── vectorTile.ts # VectorTileLike wrapper over the WASM handle
├── pkg/             # Generated by wasm-pack (gitignored)
├── dist/            # Generated by tsc (gitignored)
├── Cargo.toml
├── package.json
└── tsconfig.json
```

## Build

Requires [wasm-pack](https://rustwasm.github.io/wasm-pack/installer/) and Node.js.

```sh
# From this directory:
npm run build
```

This runs two steps in order:

1. `wasm-pack build --target bundler --out-dir pkg` — compiles the Rust crate to WASM and writes JS bindings + `.d.ts` types to `pkg/`.
2. `tsc` — compiles the TypeScript wrapper in `js/` to `dist/`.

Both steps can be run individually:

```sh
npm run build:wasm
npm run build:ts
```

## Usage

```ts
import { decodeTile } from '@maplibre/mlt-wasm';

const data = new Uint8Array(await fetch(tileUrl).then(r => r.arrayBuffer()));
const tile = decodeTile(data);

for (const [name, layer] of Object.entries(tile.layers)) {
    for (let i = 0; i < layer.length; i++) {
        const feature = layer.feature(i);
        console.log(feature.type);           // 1 | 2 | 3
        console.log(feature.id);             // number | undefined
        console.log(feature.properties);     // fetched lazily from WASM
        console.log(feature.loadGeometry()); // Point[][]
    }
}
```

## API

### `decodeTile(data: Uint8Array): VectorTileLike`

Decodes a raw MLT tile (tag-0x01 format). All column decoding happens inside WASM eagerly on this call. The returned object holds only a reference to the WASM allocation — no feature data is copied to the JS heap until accessed.

```ts
interface VectorTileLike {
    layers: Record<string, VectorTileLayerLike>;
}

interface VectorTileLayerLike {
    version: number;   // always 1 for tag-0x01
    name: string;
    extent: number;
    length: number;
    feature(i: number): VectorTileFeatureLike;
}

interface VectorTileFeatureLike {
    type: 0 | 1 | 2 | 3;
    id: number | undefined;
    extent: number;
    properties: Record<string, number | string | boolean>;
    loadGeometry(): Point[][];
}
```

### Laziness

| Access | When WASM is called |
|---|---|
| `feature.type` | Feature construction |
| `feature.id` | Feature construction |
| `feature.extent` | Layer construction (no per-feature call) |
| `feature.properties` | First access of the getter; result is cached |
| `feature.loadGeometry()` | Each call |

Features that are never accessed beyond `type` and `id` cross the WASM boundary exactly twice.

### Geometry types

MLT has six geometry types. They collapse to the four MVT wire values:

| MLT type        | `type` |
|-----------------|--------|
| Point           | 1      |
| MultiPoint      | 1      |
| LineString      | 2      |
| MultiLineString | 2      |
| Polygon         | 3      |
| MultiPolygon    | 3      |

Multi-part geometries produce multiple rings in the `Point[][]` returned by `loadGeometry()`.

### IDs

IDs are stored as 64-bit unsigned integers in MLT. They are surfaced as `f64` from WASM and returned as `number` in JS. Values above `Number.MAX_SAFE_INTEGER` (2^53 − 1) lose precision. Features without an ID column, or with a null ID value, return `undefined`.

### Properties

Only `number`, `string`, and `boolean` values are surfaced. Absent (null) optional values are omitted from the properties object entirely rather than set to `null`. Struct/shared-dictionary columns are not yet supported and are silently skipped.

## Design notes

The WASM surface is intentionally minimal: a single `MltTile` struct with flat, index-based accessors. No per-layer or per-feature WASM objects are created. The TypeScript classes `MltLayer` and `MltFeature` are pure JS — they hold only a reference to the `MltTile` WASM handle and integer indices.

Geometry is transported across the WASM boundary as a flat `Int32Array`:

```
[numRings, ring0_numPoints, x0, y0, x1, y1, …, ring1_numPoints, x0, y0, …]
```

Rings are open (no repeated closing vertex). The `decodeGeometry` function in `vectorTile.ts` reconstructs `Point[][]` with a single linear pass over this buffer.
